# LibraryOfGamecraft — Claude 運用ルール

## プロジェクト概要

Unity を使ったゲーム表現・ロジックの探求プロジェクト。
シェーダー、プロシージャル生成、カメラ、プレイヤー制御、AIなど、様々なゲーム技術をライブラリとして蓄積する。

- **エンジン**: Unity (HDRP)
- **言語**: C#
- **スクリプト配置**: `Assets/_Project/Scripts/`

---

## コーディング規約

### 再利用性の原則

- **単一責任**: 1クラス1責務。複数の役割を持たせない。
- **依存性逆転**: 具体クラスではなくインターフェースに依存する（例: `IPlayerState`, `IPlayerCommand`）。
- **ScriptableObject 活用**: 設定値・パラメータは `ScriptableObject` に外出しし、プレハブ間で共有できるようにする。
- **名前空間**: 機能カテゴリごとに名前空間を切る（例: `LibraryOfGamecraft.Terrain`, `LibraryOfGamecraft.Player`）。
- **ライブラリ化**: 汎用ロジック（ノイズ、数学ユーティリティ等）は `Scripts/Libraries/` 以下に配置し、ゲームロジックと分離する。

### 命名規則

| 対象 | 規則 | 例 |
|------|------|----|
| クラス・メソッド | PascalCase | `ChunkManager`, `GenerateMesh()` |
| フィールド（private） | `_camelCase` | `_chunkSize` |
| プロパティ（public） | PascalCase | `ChunkSize` |
| インターフェース | `I` プレフィックス | `IPlayerState` |
| イベント | `On` プレフィックス | `OnChunkLoaded` |

### コメント

- 自明なコードにコメントは不要。
- アルゴリズムの意図が不明瞭な箇所にのみ日本語でコメントを書く。
- `// TODO:` は必ず理由を添える。

## ドキュメント運用ルール

### 機能追加時

新機能を実装したら **必ず** レポートを作成する。

- **配置場所**: `Assets/_Project/Docs/`（なければ作成）
- **ファイル名**: `{機能名}_report.md`（例: `TerrainGeneration_report.md`）
- **テンプレート**: 下記参照

### バグ修正・変更時

対応するレポートが存在する場合は **必要に応じて** 更新する。

- 仕様変更・設計変更 → レポートの該当セクションを更新
- 軽微なバグ修正 → `## 変更履歴` に1行追記
- レポートがない機能への修正 → 更新不要（ただし大きな変更なら新規作成を検討）

### 課題洗い出し時

コードレビューや調査で課題・問題点を洗い出したら **必ず** 課題レポートを作成する。

- **配置場所**: `Assets/_Project/Docs/Issues/`（なければ作成）
- **ファイル名**: `{課題名}_issue.md`（例: `PlayerJump_issue.md`）
- **テンプレート**: 下記「課題レポートテンプレート」参照
- 課題を解決したら、該当レポートの `## 解決` セクションを記入し、`## ステータス` を `解決済み` に更新する。

### 実装完了チェックリスト

コードの変更・追加が完了したら、レスポンスを返す前に必ず以下を確認する。

- [ ] 新機能を追加した → `Docs/{機能名}_report.md` を作成したか
- [ ] 設計変更・仕様変更をした → 対応するレポートの該当セクションを更新したか
- [ ] 軽微な変更をした → 対応するレポートの `## 変更履歴` に1行追記したか
- [ ] 課題・問題点を洗い出した（レビュー・調査時のみ）→ `Docs/Issues/{課題名}_issue.md` を作成したか
- [ ] 既存の課題を解決した（該当する場合のみ）→ 対応する課題レポートの `## 解決` と `## ステータス` を更新したか

## レポートテンプレート

```markdown
# {機能名} レポート

## 概要

何のために作ったか、何ができるか。

## 設計

### クラス構成

| クラス | 責務 |
|--------|------|
| `FooManager` | 〇〇を管理する |

### 依存関係・使い方

どのクラスからどう使うか。コード例があれば記載。

## 実装メモ

設計上の判断、アルゴリズムの選択理由など。

## 既知の制限・TODO

- [ ] 〇〇は未対応

## 変更履歴

| 日付 | 内容 |
|------|------|
| YYYY-MM-DD | 初版作成 |
```

## 課題レポートテンプレート

```markdown
# {課題名} 課題レポート

## ステータス

未解決 / 解決済み

## 概要

どんな問題か。どこで発生しているか。

## 詳細

再現手順、影響範囲、原因の仮説など。

## 解決

どのように解決したか。変更したファイル・クラス・メソッドなど。
（解決前は空欄でよい）

## 変更履歴

| 日付 | 内容 |
|------|------|
| YYYY-MM-DD | 課題起票 |
```
